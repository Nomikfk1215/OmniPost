# OmniPost UI 迭代文档：ContentFlow 风格创作工作台

> 日期：2026-05-29  
> 文档定位：UI 迭代方案  
> 关联 Issue：`docs/ui-iteration-issue.md`  
> 参考输入：用户提供的 ContentFlow 工作台截图

## 1. 迭代定位

本轮迭代不是新增一个营销页，也不是替换产品方向，而是在 OmniPost V1 的“三栏式多平台内容创作工作台”基础上，升级为更接近真实内容生产流程的工作台 UI。

目标界面应让用户感知到：

```text
我正在编辑一篇内容
我能实时看到不同平台的发布效果
我能调整排版风格
我能确认发布平台和发布方式
我能在发布前看到检测结果
```

## 2. 截图 UI 拆解

截图中的页面可以拆成七个稳定区域：

| 区域 | 作用 | 对 OmniPost 的价值 |
|---|---|---|
| 左侧导航 | 承载产品级模块入口 | 让产品从单页 Demo 变成内容工作台 |
| 顶部操作栏 | 返回、保存状态、全平台预览、保存草稿、一键发布 | 强化工作流状态和关键动作 |
| 内容编辑区 | 标题、类型、标签、封面、正文 Markdown | 承载创作输入和内容维护 |
| 平台预览区 | 平台 Tab + 当前平台发布预览 | 展示多平台适配结果 |
| 排版样式区 | 样式、标题、正文、组件、强调 | 让用户有可控的视觉调整能力 |
| 发布设置区 | 平台选择、授权状态、发布模式、立即发布 | 完成发布前确认 |
| 内容检测区 | 错别字、敏感词、标题长度、图片检测 | 给出发布前质量反馈 |

## 3. 当前方案与目标方案差异

| 维度 | 当前 V1 工作台 | 目标 UI |
|---|---|---|
| 页面外壳 | 顶部导航为主 | 左侧导航 + 顶部操作栏 |
| 主工作区 | 左原文 / 中平台编辑 / 右预览校验 | 左编辑 / 中预览 / 右排版 / 底部发布检测 |
| 平台能力 | 平台 Tab 编辑生成内容 | 平台 Tab 预览发布效果 |
| 样式能力 | 主要依赖风格 Preset | 右侧可视化排版工具 |
| 发布确认 | 创作台内简单模拟发布 | 平台卡片 + 发布模式 + 检测结果 |
| 产品感知 | Demo 工具 | 内容运营工作台 |

需要注意：目标 UI 不应该丢掉“平台版本可编辑”能力。建议 P0 先把主体验切到截图形态，P1 再补“当前平台内容字段编辑”能力，可以通过预览区下方抽屉、右侧 Tab 或 inline editing 实现。

## 4. 设计原则

### 4.1 工作台优先

首屏直接进入可操作状态，不做大 Banner，不解释功能。页面的重点是编辑、预览、发布，而不是介绍产品。

### 4.2 状态始终可见

保存状态、平台授权、发布模式、检测结果和当前平台都需要在页面上有稳定位置。用户不应该猜“现在能不能发布”。

### 4.3 平台差异可见

公众号、知乎、B站、小红书的预览不能只是换图标和颜色，至少要在标题结构、作者信息、正文密度、图片比例、互动信息上有差异。

### 4.4 控件先轻后深

排版工具 P0 先做可选中状态和 UI 反馈，不强行变成完整编辑器。真正影响渲染的能力放到 P1，避免范围失控。

### 4.5 高密度但不拥挤

这是运营工具，不是品牌展示页。使用紧凑、清晰、可扫描的布局；避免过大的装饰、渐变背景和卡片堆叠。

## 5. 页面结构

建议桌面端布局如下：

```text
┌──────────────┬──────────────────────────────────────────────────────────────┐
│ Sidebar      │ Header: 返回 / 标题 / 保存状态 / 预览 / 保存 / 发布 / 用户   │
│              ├──────────────────────┬────────────────────────┬──────────────┤
│              │ 内容编辑区            │ 平台预览区              │ 排版样式区    │
│              │                      │                        │              │
│              ├───────────────────────────────────────────────┬──────────────┤
│              │ 发布设置区                                    │ 内容检测区    │
└──────────────┴───────────────────────────────────────────────┴──────────────┘
```

推荐尺寸：

| 区域 | 建议尺寸 |
|---|---|
| Sidebar | 220px |
| Header | 64px |
| 内容编辑区 | 主内容宽度的 42% |
| 平台预览区 | 主内容宽度的 36% |
| 排版样式区 | 主内容宽度的 22%，最小 240px |
| 发布设置区 | 占底部剩余宽度 |
| 内容检测区 | 240px 到 300px |

## 6. 模块规格

### 6.1 左侧导航

导航项：

```text
内容中心
新建内容
模板管理
平台账号
发布记录
数据统计
系统设置
```

底部版本卡片：

```text
专业版
有效期至 2026-12-31
升级版本
```

交互要求：

- 当前模块高亮。
- 图标和文字对齐稳定。
- 低高度屏幕下导航列表可滚动，版本卡片固定在底部或自然下沉。

### 6.2 顶部操作栏

左侧：

```text
返回按钮
页面标题：内容编辑
保存状态：已保存 10:30
```

右侧：

```text
预览全部平台
保存草稿
一键发布
通知
用户菜单
```

状态规则：

- 保存中：显示“保存中...”
- 已保存：显示最近保存时间。
- 保存失败：显示错误状态，并允许重试。
- 内容有未保存修改：保存草稿按钮进入强调状态。

### 6.3 内容编辑区

字段：

| 字段 | 类型 | 要求 |
|---|---|---|
| 标题 | input | 单行，支持字数统计或长度提示 |
| 内容类型 | select | 默认教程干货，可扩展测评、种草、活动、脚本 |
| 标签 | tag input | 支持新增、删除，最多建议 8 个 |
| 封面图 | upload | 支持缩略图、上传占位、删除 |
| 正文 | Markdown editor | 支持基础工具栏和字数统计 |

正文工具栏 P0：

```text
加粗 / 斜体 / 标题 / 无序列表 / 有序列表 / 链接 / 图片 / 引用 / 代码 / 撤销 / 重做
```

底部状态：

```text
Markdown 语法已开启
当前字数
编辑器设置入口
```

### 6.4 平台预览区

平台顺序：

```text
公众号 / 知乎 / B站 / 小红书
```

顶部信息：

```text
平台 Tab
AI 生成提示：以上内容为 AI 生成，您可以在下方进行编辑和调整
```

预览内容：

```text
标题
作者或账号
发布时间
阅读量或互动占位
封面图
摘要或引用块
正文
平台特色字段
```

空状态：

```text
尚未生成当前平台内容
```

加载状态：

```text
正在生成当前平台预览...
```

失败状态：

```text
生成失败，可重新生成
```

### 6.5 排版样式区

控件分组：

| 分组 | 控件 |
|---|---|
| 选择样式 | 清新简约、专业干货、轻松种草 |
| 标题样式 | 标题、副标题、小标题 |
| 正文样式 | 正文、引用、列表 |
| 内容组件 | 图片、分割线、二维码 |
| 重点强调 | 加粗、颜色、背景 |

P0 行为：

- 控件可点击、可选中。
- 状态保存到 `uiPreferences`。
- 不要求完全影响 Markdown 内容。

P1 行为：

- 标题、引用、列表、分割线等控件影响预览渲染。
- 颜色和背景只作用于预览层，不直接污染原始 Markdown。

### 6.6 发布设置区

平台卡片字段：

```text
平台图标
平台名称
授权状态：已授权 / 未授权
发布能力：可真实发布 / 模拟发布 / 模拟投稿 / 辅助发布
选中状态
```

发布模式：

```text
真实发布
模拟发布
```

默认策略：

- 默认选中模拟发布。
- 未授权平台不能选择真实发布。
- P0 可保留真实发布 UI，但真实发布按钮不调用真实平台 API。

主按钮：

```text
立即发布
```

辅助入口：

```text
定时发布
```

### 6.7 内容检测区

检测项：

```text
错别字检测
敏感词检测
标题长度检测
图片检测
```

状态：

```text
通过
警告
失败
检测中
```

规则：

- 当前平台切换后，检测结果应同步切换。
- 失败项阻止真实发布。
- 模拟发布可以继续，但要显示提示。

## 7. 状态模型建议

在现有 `WorkflowProvider` 基础上增加 UI 状态，不建议拆出独立复杂状态库。

```ts
type WorkspaceUiPreferences = {
  layoutStyle: "fresh" | "professional" | "casual";
  titleStyle: "title" | "subtitle" | "smallTitle";
  bodyStyle: "body" | "quote" | "list";
  enabledComponents: Array<"image" | "divider" | "qrcode">;
  emphasis: Array<"bold" | "color" | "background">;
};

type PublishSettings = {
  selectedPlatforms: Platform[];
  mode: "real" | "mock";
  scheduleAt?: string;
};

type PlatformAccountState = {
  platform: Platform;
  authorized: boolean;
  publishCapability: "real" | "mock" | "assist";
};
```

与现有状态的关系：

```text
rawContent
  → 生成 platformContents
  → activePlatform 决定预览和检测
  → uiPreferences 决定预览样式
  → publishSettings 决定发布行为
```

## 8. 迭代阶段

### 阶段一：视觉骨架与主布局

目标：先让 `/workspace` 长成截图中的工作台。

任务：

- 完成 Sidebar 和 Header。
- 完成五区布局。
- 内容编辑区接入现有 rawContent。
- 平台预览区复用现有 preview 组件。
- 排版样式区先做静态可交互控件。
- 发布设置和内容检测区先展示现有状态。

验收：

- 页面视觉结构与截图一致。
- 用户能完成基础内容输入和平台切换预览。
- 不破坏现有模拟发布闭环。

### 阶段二：交互打通与状态反馈

目标：让 UI 不只是静态壳，而是能反映真实状态。

任务：

- 保存草稿接入内容更新。
- 一键发布接入现有模拟发布 API。
- 检测结果接入 validators。
- 排版样式状态影响预览层。
- 预览全部平台支持弹窗或新视图。

验收：

- 用户修改内容后，保存状态变化明确。
- 发布前能看到检测结果。
- 模拟发布成功后能进入发布记录。

### 阶段三：平台化增强

目标：补齐工作台作为真实产品的连续使用体验。

任务：

- 平台账号页展示授权状态。
- 模板管理页承接排版样式和内容模板。
- 数据统计页展示发布任务和平台表现占位。
- 移动端降级布局。

验收：

- 左侧导航不再只是装饰入口。
- 用户能从内容中心继续编辑历史内容。
- 低宽度视口有可用的 Tab 式工作流。

## 9. 验收标准

功能验收：

- 可以输入标题、正文、标签和封面。
- 可以切换四个平台预览。
- 可以选择排版样式控件。
- 可以选择发布平台和发布模式。
- 可以查看内容检测结果。
- 可以触发模拟发布。

视觉验收：

- 1280px、1440px、1920px 下无横向滚动。
- 卡片、按钮、Tab、输入框状态统一。
- 文字不溢出按钮和卡片。
- 预览内容长时内部滚动，不撑坏整体布局。
- 主按钮层级明确，一键发布是页面最强动作。

技术验收：

- `npm run typecheck` 通过。
- `npm run build` 通过。
- 现有 API route 不回归。
- 现有 preview renderer 不回归。

## 11. 风险与取舍

| 风险 | 处理方式 |
|---|---|
| 页面信息密度过高 | 优先保证桌面端，移动端用 Tab 降级 |
| 排版工具范围膨胀 | P0 只做状态和视觉，P1 再影响预览 |
| 平台账号授权阻塞 | 本轮只展示授权状态，不接真实授权 |
| 平台版本编辑被弱化 | P1 增加当前平台字段编辑入口 |
| 预览和编辑状态不同步 | 统一通过 `WorkflowProvider` 管理 |

## 12. 最小可交付版本

若时间紧张，本轮至少交付：

```text
Sidebar + Header
内容编辑区
平台预览区
排版样式区静态交互
发布设置区
内容检测区
模拟发布闭环
```

可以暂缓：

```text
真实发布
定时发布
平台账号真实授权
排版样式深度改写内容
移动端完整适配
```

## 13. 总结

这次 UI 迭代的核心不是“换一套皮肤”，而是把 OmniPost 从可演示的多平台生成工具，推进到更像真实内容运营工作台的形态。

最终用户应该能在同一个屏幕里完成：

```text
写内容
看平台效果
调排版
查问题
选平台
发出去
```
